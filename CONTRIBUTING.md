# Contributing to The Next 100 Days Project

Thank you for your interest in contributing! This project thrives on community participation. Here's how you can help.

## ü§ù How to Contribute

### 1. Use and Test Tools
- Try out the tools in `tools/`
- Report bugs or suggest improvements
- Share your use cases and results

### 2. Submit Tools
- Build tools that solve real problems
- Follow the weekly theme when applicable
- Include documentation and examples

### 3. Share Patterns
- Document proven approaches in `community/patterns/`
- Include code examples and use cases
- Explain why it works

### 4. Write Case Studies
- Share your success stories in `resources/case-studies/`
- Include before/after comparisons
- Be honest about challenges

### 5. Improve Documentation
- Fix typos, clarify instructions
- Add examples and tutorials
- Translate content (if applicable)

## üìù Contribution Guidelines

### For Tool Submissions

1. **Create a new directory** in `tools/` following the naming convention:
   ```
   tools/week-XX-tool-name/
   ```

2. **Include these files:**
   - `README.md` - Tool description, usage, examples
   - Source code (organized appropriately)
   - `LICENSE` - MIT or compatible
   - `examples/` - Usage examples

3. **Documentation should include:**
   - What problem it solves
   - How to install/use
   - Code examples
   - Known limitations

### For Pattern Submissions

1. **Create a markdown file** in `community/patterns/`
2. **Follow this structure:**
   ```markdown
   # Pattern Name
   
   ## Problem
   What problem does this solve?
   
   ## Solution
   How does this pattern solve it?
   
   ## Example
   Code example here
   
   ## When to Use
   When is this pattern appropriate?
   
   ## When NOT to Use
   When should you avoid this?
   
   ## Submitted By
   Your name/GitHub handle
   ```

### For Case Studies

1. **Create a markdown file** in `resources/case-studies/`
2. **Include:**
   - Project overview
   - Challenge faced
   - Solution used
   - Results/metrics
   - Lessons learned
   - Tools/patterns referenced

## üîÑ Pull Request Process

1. **Fork the repository**
2. **Create a feature branch** (`git checkout -b feature/amazing-tool`)
3. **Make your changes**
4. **Commit with clear messages** (`git commit -m 'Add amazing tool'`)
5. **Push to your branch** (`git push origin feature/amazing-tool`)
6. **Open a Pull Request**

### PR Requirements

- Clear description of what and why
- Link to related issues (if any)
- Screenshots/examples for UI changes
- Updated documentation
- Tests (if applicable)

## üéØ Code Standards

- **Language:** Use the most appropriate language for the tool
- **Documentation:** Always include README.md
- **Examples:** Provide working examples
- **Comments:** Comment complex logic
- **Naming:** Use clear, descriptive names

## üìã Issue Reporting

When reporting issues:

1. **Check existing issues** first
2. **Use clear titles** and descriptions
3. **Include:**
   - What you expected
   - What actually happened
   - Steps to reproduce
   - Environment details
   - Screenshots (if applicable)
